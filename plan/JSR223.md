# javax.script (JSR-223) ScriptEngine — trivial compatibility shim

## Status (2026-07-11): scoped, not started

Follow-on from surveying `~/jcef/jpy` against jpype's reverse bridge (see
`plan/JepParity.md`-style survey, not written up as its own doc since this is
the only actionable item). jpy ships `org.jpy.jsr223.ScriptEngineImpl` /
`ScriptEngineFactoryImpl` so it can plug into the standard
`javax.script.ScriptEngine` API. jpype has no equivalent. User: trivial to
add on our system, worth doing purely for compatibility with tools that
discover engines via `ScriptEngineManager`/`META-INF/services`.

Note: jpy's other distinguishing feature (`PyObject.createProxy(Class)`,
adapting a `PyObject` to an arbitrary Java interface by method-name matching)
was explicitly ruled out — not needed, since assigning into a Python
variable already gets you the object as-is; no adaptation step required.

## Building blocks already present

- `org.jpype.Interpreter` (`AutoCloseable`, `getBuiltIn()`) and
  `org.jpype.SubInterpreter` — interpreter lifecycle already exists.
- `python.lang.PyBuiltIn.eval(CharSequence, PyDict globals, PyObject locals)`
  and `.exec(CharSequence, PyDict globals, PyMapping<?,?> locals)`
  (`PyBuiltIn.java:327`, `:339`) — this is the exact primitive
  `ScriptEngine.eval(String, ScriptContext)` needs to bind to.
- `python.lang.PyDict` (`PyDict.java:47`) implements `PyMapping<PyObject,
  PyObject>` with `putAll(Map<...>)`/`update(Map<...>)` — enough to build a
  `PyDict` view over a `javax.script.Bindings` (which is itself a
  `Map<String, Object>`).

## Design sketch

- New package `org.jpype.script` (or `python.script`, TBD — pick whichever
  matches the naming convention used for other integration shims, see
  `plan/Naming.md`), containing:
  - `JPypeScriptEngine implements javax.script.ScriptEngine` — wraps an
    `Interpreter`/`SubInterpreter`, translates `eval(String)` /
    `eval(Reader)` to `PyBuiltIn.eval`/`.exec`, translates
    `ScriptContext`'s `ENGINE_SCOPE`/`GLOBAL_SCOPE` `Bindings` to/from a
    `PyDict`.
  - `JPypeScriptEngineFactory implements javax.script.ScriptEngineFactory` —
    static metadata (`getEngineName()`, `getLanguageName()`, `"python"`,
    extensions `.py`, mime types `text/x-python`) plus
    `getScriptEngine()`.
  - `META-INF/services/javax.script.ScriptEngineFactory` service file
    listing `JPypeScriptEngineFactory`, so `new ScriptEngineManager()
    .getEngineByName("python")` discovers it automatically — matches jpy's
    own `Jsr223Test` pattern.
- Decide `eval` vs `exec` dispatch: JSR-223 `eval` returns a value, Python's
  `exec` doesn't — need `PyBuiltIn.eval` for expression-shaped scripts and
  fall back to `exec` (returning `null`) for statement-shaped ones, or just
  always use `exec` plus reading a conventional result variable back out of
  the bindings. Check how jpy's `ScriptEngineImpl` handles this
  (`~/jcef/jpy/src/main/java/org/jpy/jsr223/ScriptEngineImpl.java`) before
  deciding — no need to redesign this choice from scratch if jpy's approach
  is reasonable.
- `Invocable` interface (`invokeFunction`/`invokeMethod`/`getInterface`) —
  jpy implements this too; check if it's worth matching for drop-in
  compatibility with code written against jpy's engine, or if it's excess
  scope for a "just for compatibility" shim.

## Testing

- Port the shape of jpy's `Jsr223Test`
  (`~/jcef/jpy/src/test/java/org/jpy/jsr223/Jsr223Test.java`) — engine
  discovery via `ScriptEngineManager`, basic `eval(String)`, bindings
  round-trip (Java sets a binding, Python script reads it; Python script
  sets a variable, Java reads it back via `Bindings`/`get()`).

## Open questions

- Package placement (`org.jpype.script` vs `python.script`) — resolve
  against `plan/Naming.md`'s conventions at implementation time.
- Whether to target `Invocable` in the first pass or defer it.
