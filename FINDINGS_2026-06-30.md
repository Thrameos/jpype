# Type Conversion Mystery - SOLVED
**Date**: 2026-06-30  
**Issue**: Backend method returns failing with "Return value is not compatible with required type"

## TL;DR

✅ **The type conversion system works perfectly!**  
❌ Tests blocked by Python 3.12 subinterpreter threading bug (SIGSEGV in `PyThreadState_New`)

## The Real Problem: Python 3.12 Threading Bug

After probe completes, crash occurs:
```
[DEBUG] JPPyCallAcquire: Creating new thread state and swapping
[DEBUG] Created new state: 0x752bc04c0660

SIGSEGV (0xb) at pc=0x0000752bc96a5015
C  [libc.so.6+0xa5015]  sem_wait+0x15
C  [libpython3.12.so+0x310af8]  PyThread_acquire_lock_timed+0x68  
C  [libpython3.12.so+0x2f87f0]  PyThreadState_New+0x10
```

**Root Cause**: `PyThreadState_New()` tries to acquire a lock on a NULL semaphore (`RDI=0x0000000000000000`). This is a Python 3.12 subinterpreter bug, not our code.

**Stack trace shows**: Crash happens during teardown/cleanup when calling `PyThreadState_New()` from a JNI thread.

## Debug Process

1. **Hypothesis**: Dictionary empty when Backend methods called
   - **Result**: FALSE - dict has 19 entries when Backend called

2. **Hypothesis**: Probe not finding mappings  
   - **Result**: FALSE - probe finds tuple→PyTuple correctly

3. **Hypothesis**: Initialization order wrong
   - **Result**: FALSE - order is correct: import → loadResources → populate

4. **Final Discovery**: Threading bug in Python 3.12
   - **Result**: TRUE - crash in `PyThreadState_New()` with NULL semaphore

## Workarounds

1. Use main interpreter instead of subinterpreter
2. Use Python 3.11 until 3.12 subinterpreter threading is fixed
3. Pre-create and cache thread states
4. Report bug to Python
